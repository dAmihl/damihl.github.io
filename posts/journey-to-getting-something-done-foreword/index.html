Journey to Getting Something Done - Foreword

January 9, 2020

I have a strong history of successfully not getting things done. In fact theres a very specific set of things that I'm not getting done: Personal stuff.

Obviously I don't struggle to get things done at work, otherwise I wouldn't have a job.
It's also no problem for me to finish a box of chocolates, or binge watch an entire season of Brooklyn 99 the day it releases on Netflix. No, it's this very specific set of things that just refuse to get to a state which most people would call ‘Done’.

While I'm sure there are many people with that same level of expertise in not getting things done it is in no way a particularly helpful skill to have.

The thing is: You will never hear about people who don't get things done - well, maybe if you are really, extraordinarily good at it.

I like to imagine that there have been numerous attempts to get known by not gettings things done but no one went through with it yet.

Of course you can be famous for not getting things done but only in a negative way - probably no one will tell you: “I admire you and your ability to not get things done!”

I have started multiple prototypes for game ideas that never made it past the “base mechanic's working, good enough for today” phase. Although I feel like I want to have stuff to show I'm struggling to get that stuff to a state where it can be shown.

So there's my problem and I think there are multiple reasons for this:

  • First, I tend to slack when theres nothing at stake. I mean, it's just a side project - it's not that important, right?
  • Obviously time is a big factor. There's work to be done, dishes to be cleaned, friends and family to be looked after, couches to be relaxed on, series to be binged, .. whoops, there we go. Back to Point 1 again.
  • Scope and overengineering. I feel like this is widely known but nonetheless a very important one. Let's be honest, it's really hard to keep a small, doable scope for hobby projects. Thinking about making a small webapp? Better check out all available frameworks, new technologies, make it reliable, modular, extendable. Want to make a tetris clone? Better make it playable on all platforms with progression and meta-progression, RPG elements, maybe some TCG mechanics. You know what would be fun? BattleRoyale Mode! (I know, Tetris BR is a thing) Before you even started to program you are already overwhelmed. Also after coming up with all these ideas you deserve a rest on the couch and maybe watch the next episode of The Witcher.. whoops, back to Point 1 again.

So how should I face this issue? Well, if you think about, getting something done seems like a pretty good step in the right direction.

And that's what I'm going to do and what this post and all following (hopefully, see first sentence :)) are really about:

I will create a simple game from start to finish.

At this point I have to admit that this is not the first time I'm saying this sentence. Few years ago I felt exactly the same. I decided that over the summer break (I was in university still) I would create a game (with the help of my friend who created all assets) from start to finish in the given time frame and I will release it before Uni starts again, no matter what. And I'm proud to say that it worked! (if you're interested you can check it out here: “Elements” Play Store Link) What I'm not so proud to say is that this was the only time that it worked. Since then I haven't got anything near the finish line again.

The Plan

I decided that I will define some rules and constraints in a game-jamy fashion. This should help to stay in a doable scope and make it a little easier. There will be no timeframe as, still, it's a side project and burning out on finishing something isn't a very nice goal either. It should stay fun which should be possible by constraining the scope of the game, thus increasing the sense of progress, while not turning it into a second day job.

Here's what I'm planning to do:

Think of the simplest game you can imagine, try to spice it up a little, and finish it.

My Constraints

  • The game only consists of 3 “Screens”: A “Menu” screen, a “Game” screen, and a “Game Over” screen. Options-Screen and all the other stuff should not really be needed when considering all other constraints.
  • The game focuses to be playable on a single static platform.
  • It should only consist of a single main mechanic. The mechanic can be altered to spice it up, but it should stay the only one.
  • Assets should not take longer than, say, 5 hours to create per asset. If it doesn't look good, worry about it until after the game is “finished”.

In the following posts I will try to document the different stages that I am at in development, kind of like a devlog I guess. I hope that maintaining a series of blog posts will also increase the chances of me finishing this experiment.

In the rare chance that you like the idea and want to continue reading, and in the not-so-rare chance that the series stops somewhere in the middle, please give me a poke, send me a message and help me get back on this journey to getting something done :)
